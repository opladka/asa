UNIT 01 Foundations & Data Modelling

A database is an organized, shared collection of logically related data and metadata, managed by a DBMS. It supports multiple users, enforces integrity, and provides concurrent access.

File System: data redundancy, no relationships, manual integrity, single-user
Database: minimal redundancy, relationships enforced, integrity constraints, multi-user

Key differences: DBMS offers data abstraction, query language support, transaction management, and security control — none of which a flat file system provides natively.

Database Administrator (DBA) — manages schema, security, performance, backups.

Database Designer — defines structure, entities, relationships, constraints.

Application Programmers — write programs using DML embedded in host language.

End Users — Naïve (via forms/apps), Casual (ad-hoc queries), Sophisticated (power analysts), Standalone (personal DB).

Metadata is "data about data" — it describes the structure, types, constraints, and relationships of actual data, stored in the data dictionary / system catalog.

It is important because the DBMS uses it to validate queries, enforce constraints, optimize execution, and provide data independence.

Data redundancy means storing the same data in multiple places. It leads to inconsistency, wasted storage, and update anomalies.

DBMS controls it through normalization, centralized schema definition, and enforcing referential integrity — so each fact is stored once and referenced by all who need it.

Data integrity ensures accuracy and consistency of data over its lifecycle. Types:

• Entity Integrity — every row has a unique, non-null primary key.
• Referential Integrity — foreign key values must match existing primary keys.
• Domain Integrity — values must conform to defined type/range.
• User-defined Integrity — business rules enforced via CHECK/TRIGGER.

1. Self-describing nature — catalog stores schema alongside data.
2. Insulation between programs and data — data independence.
3. Data abstraction — conceptual view hides physical details.
4. Multiple views — different users see different subsets.
5. Data sharing & multiuser transaction processing — concurrency control.
6. Controlling redundancy — central schema eliminates duplication.
7. Enforcing integrity constraints, Security restrictions, Backup & Recovery.

DBMS overhead may be unjustified when:

• Data is simple, well-defined, and won't change.
• Only a single user needs access.
• Real-time embedded system with no multi-user need.
• Organization cannot afford DBMS licensing/maintenance.
• Application needs extreme speed with no concurrency (e.g., sensor logging).

A DBMS contains not only the data itself but also a system catalog (data dictionary) that stores the complete definition (schema) of the database structure.

This allows the DBMS to work with different databases without needing hard-coded schema information — unlike traditional file-processing where programs must know file format in advance.

Program-data independence means the application programs do not depend on how data is physically stored. You can change storage format, file organization, or access paths without rewriting application code.

This is achieved through the DBMS's abstraction layer — programs interact with the logical schema, not the physical storage.

A data model is a set of concepts used to describe the structure, relationships, and constraints of a database.

Types: High-level (Conceptual) — ER model; Representational (Implementation) — Relational, Network, Hierarchical; Low-level (Physical) — describes how data is stored on disk.

Schema = structure (blueprint) · rarely changes
Instance = actual data at a point in time · changes often

The schema is the overall design — table names, columns, types, constraints. An instance (also called database state or extension) is the set of data stored at a particular moment.

External/View level — what individual users see (user views).
Conceptual/Logical level — describes what data is stored and relationships; hides physical details.
Internal/Physical level — how data is actually stored, indexes, file organization.

Logical Data Independence — ability to change the conceptual schema without changing external schemas/application programs. Harder to achieve.

Physical Data Independence — ability to change the internal schema (storage) without changing the conceptual schema. More commonly achieved in most DBMS.

The three-schema architecture (ANSI/SPARC) separates the database into external, conceptual, and internal levels. It was proposed to achieve:

• Data independence (logical & physical)
• Separation of user views from physical storage
• Support for multiple user perspectives simultaneously
• Portability across different hardware/OS platforms

Entity — object with independent existence (rectangle).
Attribute — property of entity (ellipse). Types: simple, composite, multivalued (double ellipse), derived (dashed).
Relationship — association between entities (diamond).
Weak Entity — cannot be identified alone; depends on owner (double rectangle).

A weak entity does not have a key attribute of its own; it is identified by its relationship to an owner (strong) entity plus its partial key (discriminator).

Example: DEPENDENT is weak; identified by (Employee_SSN + Dependent_Name). A strong entity has its own primary key and exists independently.

1:1 — one entity on each side (e.g., Employee MANAGES Department).
1:N — one on left, many on right (e.g., Department HAS Employees).
M:N — many on both sides (e.g., Student ENROLLS IN Courses).

Cardinality constrains how many instances of one entity relate to instances of another.

Specialization is a top-down process — starting with a general entity (superclass) and defining sub-entities (subclasses) with unique attributes. E.g., EMPLOYEE → ENGINEER, MANAGER.

Generalization is the bottom-up reverse — combining similar entities into a general superclass. E.g., CAR + TRUCK → VEHICLE.

Specialization = superclass → subclasses (top-down)
Generalization = subclasses → superclass (bottom-up)

Aggregation treats a relationship set (and its participating entities) as a higher-level entity, allowing it to participate in another relationship.

It is needed when a relationship itself needs to have a relationship with another entity — which cannot be directly represented in standard ER. Shown as a rectangle around the relationship diamond.

Disjoint (d) — an entity can belong to at most one subclass at a time (mutually exclusive). E.g., EMPLOYEE is either HOURLY or SALARIED.
Overlapping (o) — an entity can belong to multiple subclasses simultaneously. E.g., a STUDENT can be both GRADUATE_STUDENT and TEACHING_ASSISTANT.

An M:N relationship is mapped by creating a new cross-reference (junction/associative) relation. This new relation includes:

• Primary keys of both participating entity relations as foreign keys.
• Both FK together form the primary key of the junction table.
• Any attributes of the relationship are also added to this table.

STUDENT(sid) + COURSE(cid) → ENROLL(sid, cid, grade)

DDL (Data Definition Language) — defines schema: CREATE, ALTER, DROP, TRUNCATE.
DML (Data Manipulation Language) — manipulates data: SELECT, INSERT, UPDATE, DELETE.
DCL (Data Control Language) — controls access: GRANT, REVOKE.

DELETE — DML, removes rows with WHERE condition, can be rolled back, triggers fire, slower.
TRUNCATE — DDL, removes ALL rows quickly, cannot be rolled back (in most DBMS), no WHERE, resets identity.
DROP — DDL, removes the entire table/object including structure and data permanently.

PRIMARY KEY — unique + not null, one per table.
FOREIGN KEY — references PK of another table, enforces referential integrity.
UNIQUE — no duplicate values, allows one NULL.
NOT NULL — column must have a value.
CHECK — value must satisfy a Boolean expression: CHECK(age >= 18).
DEFAULT — assigns a value when none is provided.

A VIEW is a virtual table defined by a SELECT query. It has no physical storage.

Advantages: data security (hides columns), logical data independence, simplifies complex queries, tailors user perspectives.

Limitations: cannot always be updated (non-updatable views); performance overhead since view query is re-executed each time; cannot have indexes directly (unless materialized).

An index is a data structure (like a B+ tree) that speeds up data retrieval at the cost of extra storage and slower writes.

Types: Clustered (data rows sorted by index key, one per table) vs Non-clustered (separate structure, many allowed). Also: Unique, Composite, Full-text.

CREATE INDEX idx_name ON table(column);

PRIMARY KEY: Only one per table; cannot contain NULL values; automatically creates a clustered index.

UNIQUE: Multiple per table; allows one NULL (in most DBMS); creates a non-clustered index by default. Both enforce uniqueness but PK is the main identifier.

ALTER TABLE modifies an existing table's structure without losing data.

ALTER TABLE emp ADD salary DECIMAL(10,2);
ALTER TABLE emp MODIFY name VARCHAR(100);
ALTER TABLE emp DROP COLUMN age;
ALTER TABLE emp ADD CONSTRAINT fk_dept FOREIGN KEY(dno) REFERENCES dept(dno);

A database language allows users to define, manipulate, and control data. SQL combines:

• DDL — schema definition
• DML — data retrieval & modification
• DCL — access control
• TCL — transaction control (COMMIT, ROLLBACK, SAVEPOINT)

UNIT 02 Relational Model & Advanced SQL

Relation — a table with rows and columns.
Tuple — a single row in the relation.
Attribute — a column; has a domain (allowed values).
Degree — number of attributes (columns).
Cardinality — number of tuples (rows).

Relational Algebra is a procedural query language operating on relations and returning relations.

Fundamental: σ (Select), π (Project), ∪ (Union), − (Set Difference), × (Cartesian Product), ρ (Rename).

Derived: ⋈ (Join), ∩ (Intersection), ÷ (Division).

Relational Algebra is procedural — specifies HOW to get data (sequence of operations).

Relational Calculus is declarative — specifies WHAT data to get without describing procedure. Two types: Tuple Relational Calculus (TRC) and Domain Relational Calculus (DRC).

Two relations must be union-compatible:

1. Same number of attributes (same degree).
2. Corresponding attributes have the same domain.
This also applies to INTERSECTION and SET DIFFERENCE.

Referential Integrity: A FK value in a child table must either be NULL or match an existing PK value in the referenced (parent) table.

Violation actions on parent row DELETE/UPDATE: RESTRICT/NO ACTION, CASCADE (propagate change), SET NULL, SET DEFAULT.

Superkey — any set of attributes that uniquely identifies tuples (may include extra attributes).
Candidate Key — minimal superkey (no unnecessary attributes). A relation may have multiple.
Primary Key — the chosen candidate key used as the main unique identifier; must be NOT NULL.

Aggregate functions operate on sets of values and return a single value:

COUNT() — number of rows; SUM() — total of numeric column; AVG() — average; MIN() / MAX() — extremes.

Note: COUNT(*) counts all rows including NULLs; COUNT(col) ignores NULLs.

GROUP BY groups rows sharing the same value for the given column(s) so aggregates can be computed per group.

HAVING filters groups (like WHERE but for aggregated results).

SELECT dept, COUNT(*) AS cnt
FROM employee
GROUP BY dept
HAVING COUNT(*) > 5;

Execution order: WHERE → GROUP BY → HAVING → SELECT → ORDER BY.

A sub-query is a SELECT nested within another SQL statement. It executes once and passes result to the outer query.

A correlated sub-query references columns from the outer query — it is re-evaluated for EACH row of the outer query, making it dependent and usually slower.

-- Correlated: finds employees earning more than dept avg
SELECT name FROM emp e1
WHERE salary > (SELECT AVG(salary) FROM emp e2 WHERE e2.dno=e1.dno);

INNER JOIN — returns only matching rows from both tables.
LEFT (OUTER) JOIN — all rows from left + matching from right; NULLs where no match.
RIGHT (OUTER) JOIN — all rows from right + matching from left.
FULL (OUTER) JOIN — all rows from both; NULLs where no match on either side.
CROSS JOIN — Cartesian product (all combinations).
SELF JOIN — table joined with itself using aliases.

ANY / SOME — TRUE if the condition is satisfied for at least one value returned by subquery. E.g., sal > ANY(subquery) means sal > minimum of subquery.
ALL — TRUE only if condition holds for every value. sal > ALL(subquery) means sal > maximum.
EXISTS — TRUE if subquery returns at least one row; used to test for existence, typically with correlated sub-queries.

WHERE — filters individual rows BEFORE grouping; cannot use aggregate functions.
HAVING — filters groups AFTER GROUP BY; can use aggregate functions.

WHERE filters rows · HAVING filters groups

UNION — combines results of two SELECT queries, removes duplicates (use UNION ALL to keep).
INTERSECT — returns only rows present in BOTH results.
EXCEPT / MINUS — returns rows in first result not in second.
All require union-compatible queries (same number and type of columns).

COMMIT — permanently saves all changes made in the current transaction. After commit, changes cannot be rolled back.

ROLLBACK — undoes all changes since the last COMMIT or SAVEPOINT, restoring the database to a consistent state.

SAVEPOINT — creates a named intermediate point within a transaction. ROLLBACK TO savept undoes changes only to that point.

UNIT 03 Normalization, Transactions & Concurrency

A Functional Dependency (FD) X → Y means: for any two tuples, if they agree on X, they must agree on Y. X functionally determines Y.

Types: Full FD — Y depends on all of X (not a proper subset); Partial FD — Y depends on a proper subset of X; Transitive FD — X→Z through X→Y→Z where Y is not a key.

A relation is in 1NF (First Normal Form) if all attribute values are atomic (indivisible) and there are no repeating groups or multi-valued attributes.

To convert: replace multi-valued attributes with separate rows or a separate table; ensure each cell holds a single value; add a primary key.

A relation is in 2NF if it is in 1NF AND every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies).

Fixes: Insertion anomaly (can't insert fact without another), Update anomaly (must update many rows), Deletion anomaly (deleting one fact deletes another).

Only relevant when the PK is composite. Remove partial dependencies by decomposing into separate relations.

A relation is in 3NF if it is in 2NF AND there are no transitive dependencies — no non-key attribute depends on another non-key attribute.

3NF: For every FD X→Y, either X is a superkey, or Y is part of some candidate key.

2NF removes partial dependencies; 3NF additionally removes transitive dependencies of non-prime attributes on non-key attributes.

BCNF (Boyce-Codd Normal Form) is a stronger version of 3NF: for every non-trivial FD X→Y, X must be a superkey.

A relation satisfies 3NF but NOT BCNF when a non-prime attribute functionally determines part of a candidate key, i.e., when there are overlapping candidate keys where a determinant is not a superkey.

BCNF decomposition always achieves lossless join but may not preserve all FDs.

A decomposition of R into R1 and R2 is lossless if joining R1 ⋈ R2 reproduces exactly R with no spurious tuples.

Condition (for binary decomposition): the common attributes of R1 and R2 must form a superkey of R1 OR R2. This is tested using the FD-based test: R1∩R2 → R1 − R2 or R1∩R2 → R2 − R1.

A decomposition is dependency-preserving if every functional dependency in the original relation F can be checked in one of the decomposed relations, i.e., the union of FDs projected onto each sub-relation implies all original FDs.

BCNF may not preserve dependencies; 3NF decomposition always can.

4NF — addresses Multivalued Dependencies (MVDs). A relation is in 4NF if for every non-trivial MVD X ↠ Y, X is a superkey. MVDs occur when one attribute in a table uniquely determines multiple independent sets of another attribute.

5NF (PJNF) — addresses Join Dependencies. A relation is in 5NF if every join dependency is implied by candidate keys. Very rarely applied in practice.

Atomicity — all or nothing; either all operations complete or none take effect (handled by recovery).

Consistency — transaction takes DB from one valid state to another, preserving all constraints.

Isolation — concurrent transactions appear to execute serially; intermediate state not visible to others (handled by concurrency control).

Durability — once committed, changes persist even after system failure (handled by recovery/logging).

Active → Partially Committed → Committed
Active → Failed → Aborted
Partially Committed → Failed (on write error)

Active: transaction is executing.
Partially Committed: last statement executed; changes in buffer.
Committed: changes permanently written to disk.
Failed: error detected, cannot proceed.
Aborted: rolled back; DB restored to pre-transaction state.

A concurrent schedule is serializable if it produces the same result as some serial execution of the same transactions.

Conflict Serializable: can be transformed to a serial schedule by swapping non-conflicting operations. Tested via a precedence graph — acyclic = conflict serializable.

View Serializable: broader notion — same reads, same writes as some serial schedule. Every conflict-serializable schedule is view-serializable but not vice versa.

Shared Lock (S / Read Lock) — allows concurrent reads but not writes. Multiple transactions can hold S-locks on the same item simultaneously.

Exclusive Lock (X / Write Lock) — allows read and write but no other lock (S or X) can coexist. Only one transaction holds X-lock.

Two-Phase Locking (2PL): Growing phase (acquire locks) → Shrinking phase (release locks). 2PL ensures conflict serializability.

Each transaction T is given a unique timestamp TS(T) when it starts. Each data item Q maintains read_TS(Q) and write_TS(Q).

Rules: A READ by T is rejected if TS(T) < write_TS(Q) (Q was already written by a younger transaction). A WRITE is rejected if TS(T) < read_TS(Q) or TS(T) < write_TS(Q). Rejected transactions are rolled back and restarted with a newer timestamp.

A deadlock occurs when two or more transactions wait for each other to release locks, creating a cycle. None can proceed.

Detection: Maintain a wait-for graph; a cycle indicates deadlock. Run periodically.

Resolution: Select a victim transaction (lowest cost), roll it back and release locks. Prevention techniques: Wait-Die, Wound-Wait, No-wait, Timeout.

Recoverable Schedule: If transaction T reads a value written by T', then T' must commit before T commits. Otherwise, if T' aborts after T commits, T has read a "dirty" uncommitted value.

Cascadeless (Avoids Cascading Rollback): A transaction can only read values written by committed transactions. Stricter than recoverable; prevents "cascading abort" ripple effect.

A stored procedure is a precompiled set of SQL statements stored in the database and executed by name. It can accept parameters and return values.

Advantages: Reduced network traffic (one call, many statements); precompiled = faster execution; code reusability; centralized business logic; security (users call procedure, not tables directly).

CREATE PROCEDURE proc_name(param TYPE) AS BEGIN ... END;

A trigger is a named database object that automatically executes (fires) in response to a specified event on a table or view.

Components: Event (INSERT/UPDATE/DELETE), Timing (BEFORE/AFTER/INSTEAD OF), Level (ROW or STATEMENT), Condition (optional WHEN clause), Action (PL/SQL body).

Function: Must return a value; can be used in SELECT; cannot modify database state (in most DB); called in expressions.
Stored Procedure: May or may not return values (via OUT params); cannot be used directly in SELECT; can perform DML; called with EXECUTE/CALL.

The Write-Ahead Log (WAL) records all changes to the DB before they are applied to disk. Each log record contains: transaction ID, data item, old value (for UNDO), new value (for REDO).

Recovery: REDO committed transactions whose changes may not have reached disk; UNDO uncommitted transactions that were active during crash.

A checkpoint is a snapshot point where the DBMS flushes all dirty (modified) buffers to disk and records the current transaction list in the log.

During recovery, only transactions after the last checkpoint need to be examined — reducing recovery time significantly. Checkpoints limit how far back REDO/UNDO operations must go.

Granularity refers to the size of data items on which locks are set: Field → Record (Row) → Page → Table → Database.

Fine granularity = more parallelism but high overhead (many locks). Coarse granularity = less overhead but less concurrency.

Multigranularity locking uses intention locks (IS, IX, SIX) at higher levels to indicate intent to lock lower levels, allowing efficient compatibility checking without traversing the whole hierarchy.

UNIT 04 File Structures, Indexing & Advanced DBMS

A heap file (unordered file) stores records in insertion order with no particular arrangement. New records are appended at the end.

Pros: Fast insertion O(1). Cons: Linear scan for search O(n). Used when: data is frequently inserted and bulk-scanned, or when queries don't filter on a specific key.

A sorted file maintains records in sorted order on an ordering field. Enables binary search — O(log n) for search.

Advantages: Fast range queries, efficient sequential access, supports ordered output without sorting. Disadvantages: Costly insertions (maintain sort order), may need overflow pages.

Hashing applies a hash function h(key) to determine the bucket where a record is stored. O(1) average access time.

Static Hashing: Fixed number of buckets M; h(k) = k mod M. Problem: overflow when data grows. Dynamic Hashing (Extendible/Linear): Adjusts number of buckets as data grows, avoiding overflow by splitting or expanding bucket directory.

A collision occurs when two different keys hash to the same bucket.

Resolution methods: Open Addressing (linear/quadratic probing — find next empty slot), Chaining (linked list of overflow records per bucket), Overflow Pages (linked extra pages to the full bucket).

Open — prepare file for access (load header, set pointer).
Close — flush buffers, release resources.
Reset — move pointer to start for re-reading.
Find — locate first record satisfying condition.
Read — retrieve current record.
FindNext — locate next matching record.
Insert / Delete / Modify — write operations.

A single-level index is a sorted file of (key, pointer to record) pairs. Binary search on index is faster than linear search on data file.

Types: Primary Index — on ordered key field of sorted file; one entry per block (sparse). Clustering Index — on non-key ordering field; one entry per distinct value. Secondary Index — on non-ordering field; dense (one entry per record).

When the primary index itself becomes too large for binary search to be efficient, a multilevel index treats the first-level index as a data file and builds another index over it — recursively, until the top level fits in one disk block.

This is the idea behind B-tree and B+ tree structures which self-maintain balance as data changes.

A B-tree of order m: each node has up to m-1 keys and m pointers; balanced (all leaves at same level); data pointers at every node (internal and leaf).

A B+ tree: data pointers ONLY at leaf nodes; internal nodes store only keys for routing; all leaves linked in a doubly linked list for efficient range queries. B+ trees are preferred in DBMS because sequential access is much faster.

B-tree: data at every node · B+ tree: data only at leaves, leaves linked

Internal node: at most p tree pointers; at most p-1 keys; at least ⌈p/2⌉ pointers (except root).

Leaf node: at most p-1 (key, data pointer) pairs; at least ⌈(p-1)/2⌉ pairs; one pointer to next leaf.

Root: at least 2 pointers (if not leaf). Height h ≤ log⌈p/2⌉(N) where N is number of keys.

Dense Index — one index entry for every record in the data file. Faster search but more storage. Used for secondary indexes.

Sparse Index — one entry per disk block (or per distinct value). Smaller index; requires the data file to be sorted on the index key. Primary indexes are usually sparse.

Secondary storage (magnetic disk, SSD, HDD) provides persistent, large-capacity storage. Data is organized in blocks/pages (typical 4KB–16KB).

Disk access time = Seek time (move head to track) + Rotational delay (wait for block) + Transfer time (read/write block). Minimizing disk I/O is the primary goal of file organization and indexing strategies.

An OODBMS stores data as objects (as in OOP), supporting complex data types, inheritance, encapsulation, and methods alongside data.

RDBMS stores data in tables (rows/columns), uses SQL, handles simple types well. OODBMS is better for CAD, multimedia, engineering apps. RDBMS wins for business/financial data needing ACID, joins, and ad-hoc queries.

• Object identity (OID) — each object has a unique, system-generated ID.
• Encapsulation — data and methods bundled together.
• Inheritance — classes inherit attributes/methods from superclasses.
• Polymorphism — same operation behaves differently in different subclasses.
• Complex objects — support for nested/composite types.
• Persistence — objects persist beyond program execution.

A DDBMS is a collection of logically interrelated databases distributed over a computer network, managed so they appear as a single database to the user.

Key properties: Local autonomy, No reliance on a central site, Continuous operation, Location independence, Fragmentation independence, Replication independence.

Fragmentation divides a relation into pieces stored at different sites.

Horizontal Fragmentation — each fragment is a subset of rows (σ condition).
Vertical Fragmentation — each fragment is a subset of columns (π columns); must include primary key in each.
Mixed/Hybrid — combination of horizontal and vertical.

Advantages: Improved reliability (no single point of failure), better performance (local access), scalability, data sharing across locations.

Challenges: Complexity of distributed query optimization, network dependency, distributed transaction management (2-phase commit protocol), maintaining consistency across replicas, higher overhead for joins across sites, security across multiple nodes.
