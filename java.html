Unit 01

JVM & Java Architecture

1.1 Overview & Characteristics

What is Java?

Java is a high-level, object-oriented, platform-independent programming language developed by Sun Microsystems (now Oracle) in 1995. It was designed with the principle of "Write Once, Run Anywhere" (WORA), meaning compiled Java code can run on any machine that has a Java Virtual Machine.

The key characteristics of Java that distinguish it from other languages are:

Platform Independent
Java source code compiles to bytecode (.class files), not native machine code. The JVM on each OS interprets this bytecode, so the same program runs on Windows, Linux, macOS, etc.
Object-Oriented
Everything in Java revolves around objects and classes. The four pillars — Encapsulation, Inheritance, Polymorphism, Abstraction — are core to the design.
Robustness
Java eliminates pointers (direct memory access), provides automatic garbage collection, strong type-checking at compile time, and runtime exception handling.
Secure
Java runs inside a sandboxed JVM environment. The bytecode verifier, security manager, and class loaders together prevent malicious code execution.
Multithreaded
Java has built-in language-level support for multithreading, allowing concurrent task execution within a single program.
Distributed
Java supports distributed computing via RMI (Remote Method Invocation) and socket-based networking out of the box.
Dynamic
Java programs can load classes at runtime dynamically. New classes can be loaded without recompiling the program.
High Performance
While interpreted, the JIT (Just-In-Time) compiler compiles frequently executed bytecode to native code at runtime, dramatically improving performance.
1.2 Compilation & Execution Process

From Source Code to Execution

Java's two-phase compilation model is the foundation of its platform independence. Understanding each phase is critical.

Phase 1 — Compilation

The Java source file (MyProgram.java) is processed by javac, the Java compiler. It performs lexical analysis, parsing, semantic analysis, and produces platform-neutral bytecode stored in MyProgram.class. This bytecode is a compact set of instructions for the JVM, not the real CPU.

Phase 2 — Execution

The java command launches the JVM, which: (1) uses the Class Loader subsystem to locate and load the .class file; (2) runs the Bytecode Verifier to ensure the bytecode is valid and safe; (3) interprets bytecode instruction by instruction. The JIT compiler monitors "hot spots" (frequently executed code) and compiles them to native machine code for speed.

# Step 1: Compile source to bytecode
javac HelloWorld.java         # produces HelloWorld.class

# Step 2: Run with JVM
java HelloWorld               # JVM interprets HelloWorld.class

# View bytecode disassembly
javap -c HelloWorld           # shows bytecode instructions
1.3 Organization of the JVM

JVM Internal Architecture

The JVM is a complex subsystem with multiple components working together. Its major sub-systems are:

Class Loader Subsystem
Loads, links, and initializes classes
Runtime Data Areas
Method Area, Heap, JVM Stack, PC Register, Native Stack
Execution Engine
Interpreter + JIT Compiler + GC
Native Method Interface
Bridges Java with native libraries (JNI)
Method Area (Class Area)
Stores per-class data: runtime constant pool, field/method data, bytecode of methods, and static variables. Shared across all threads. Also called the "PermGen" (pre-Java 8) or "Metaspace" (Java 8+).
Heap
The runtime data area from which memory for all class instances (objects) and arrays is allocated. There is only one heap per JVM instance. The Garbage Collector manages this area.
Java Stack (JVM Stack)
Each thread gets its own JVM Stack, created when the thread starts. It holds frames — one per method call. A frame stores local variables, operand stack, and a reference to the runtime constant pool.
PC Register
Each thread has its own Program Counter Register, which holds the address of the currently executing JVM instruction. It's undefined if the current method is native.
Native Method Stack
Supports native (non-Java) methods written in C/C++ that are called via JNI. Separate from the JVM Stack.
1.4 JVM as Interpreter & Emulator

How the JVM Interprets Bytecode

The JVM acts as a software emulator of an idealized machine. Rather than running on real hardware directly, bytecode instructions are executed by the JVM's execution engine.

The JVM is a stack-based virtual machine: most operations push and pop values on the operand stack within each stack frame. This is unlike register-based architectures (like x86). This design keeps bytecode compact and portable.

Interpreter vs JIT Compiler

The interpreter translates each bytecode instruction to native code one at a time on every execution — simple but slow for loops. The JIT compiler profiles the running program and compiles frequently executed "hot" bytecode paths to native machine code, caching the result. Future calls execute native code directly, giving near-native performance.

The HotSpot JVM uses a tiered compilation strategy: C1 compiler for quick startup optimization, C2 compiler for aggressive long-running optimization.

1.6 Class File Format & Verification

The .class File Structure

Every compiled Java class produces a .class file with a precisely defined binary format. It begins with the magic number 0xCAFEBABE followed by version information, and then a structured sequence of tables.

Magic Number (0xCAFEBABE)
4 bytes that identify the file as a valid Java class file. The JVM checks this first.
Minor/Major Version
Indicates the Java version that compiled the class (e.g., major version 61 = Java 17). JVM rejects class files with higher versions than it supports.
Constant Pool
A table of symbolic references: class names, method names, field names, string literals, integer/float constants. All other structures in the file reference the constant pool by index.
Access Flags
Bit mask indicating properties: ACC_PUBLIC, ACC_FINAL, ACC_INTERFACE, ACC_ABSTRACT, ACC_SYNTHETIC, etc.
Fields Table
Describes all fields (instance and static) with their names, descriptors (type signatures), and attributes.
Methods Table
Describes all methods: name, descriptor, access flags, and a Code attribute containing the actual bytecode, exception tables, local variable tables, and line number tables.
Attributes
Extensible metadata: SourceFile, InnerClasses, EnclosingMethod, AnnotationDefault, StackMapTable (for faster verification in Java 7+).

Bytecode Verification

Before execution, the JVM's Bytecode Verifier performs a series of checks to ensure the class file is structurally valid and will not violate the JVM's security guarantees. It checks: (1) The class file has proper format and valid constant pool; (2) All opcodes have valid operands; (3) No stack underflow/overflow; (4) Type safety — no integer is used as a pointer; (5) Field accesses obey access control rules; (6) Local variable types are consistent. The StackMapTable attribute (Java 7+) enables faster "type-checking" verification instead of "type-inference" verification.

1.7 Memory & Garbage Collection

JVM Memory Management

Java automates memory management through the Garbage Collector (GC). The programmer allocates objects with new; the GC automatically frees memory when objects are no longer reachable.

Heap Structure (Generational GC)

The heap is divided into generations based on the observation that most objects die young (weak generational hypothesis):

Young Generation
New objects are allocated here. Divided into Eden and two Survivor spaces (S0, S1). Minor GC is fast and frequent here. Objects that survive multiple GC cycles are promoted to Old Gen.
Old (Tenured) Generation
Stores long-lived objects that survived Young Gen GC. Major GC (Full GC) is slower and less frequent.
Metaspace (Java 8+)
Replaced PermGen. Stores class metadata. Uses native memory (not heap), grows automatically by default.

GC Algorithms

  • Serial GC — Single-threaded, for small apps. Stops all threads (Stop-the-World).
  • Parallel GC — Multi-threaded GC, better throughput for multi-core systems.
  • CMS (Concurrent Mark Sweep) — Concurrent phases to minimize pause times. Deprecated in Java 9.
  • G1 (Garbage First) — Default since Java 9. Divides heap into equal-sized regions; collects regions with most garbage first. Predictable pauses.
  • ZGC / Shenandoah — Ultra-low latency GCs with sub-millisecond pauses for large heaps.
Java Stack frames are automatically released when a method returns — only heap objects require GC. StackOverflowError occurs when the call stack exceeds its limit (infinite recursion).
1.8 JVM Security Promises

Security Architecture

Java's security model is multi-layered. The JVM provides a foundation of security guarantees that higher-level mechanisms build upon.

Type Safety
The bytecode verifier ensures that code cannot forge object references, access memory illegally, or violate type rules — eliminating entire classes of buffer overflow attacks common in C/C++.
No Pointer Arithmetic
Java has no raw pointers. You cannot compute an address and dereference it arbitrarily. All access is through references that the JVM controls.
Array Bounds Checking
Every array access is bounds-checked at runtime. Out-of-bounds access throws ArrayIndexOutOfBoundsException rather than corrupting memory.
Security Manager (legacy)
Allowed fine-grained access control: file system, network, reflection, system exit, etc. Deprecated in Java 17, removed in Java 24.
Security Policy Files
Policy files define permissions granted to code from specific sources (codebase URLs or signed JARs). E.g., an applet from www.example.com may only read temp files, not write to the OS.
1.9 Class Loaders & Sandbox Model

Class Loader Hierarchy

Class loaders are the gatekeepers of Java's security. They load class bytecode into the JVM and define class namespaces. Java uses a delegation hierarchy:

Bootstrap Class Loader
Written in native code (C++). Loads core Java API classes from rt.jar / the bootstrap modules (java.lang, java.util, etc.). The parent of all class loaders. Cannot be instantiated.
Extension / Platform Class Loader
Loads classes from the JDK extension directory (jre/lib/ext) or java.ext.dirs system property. Delegates to Bootstrap first.
Application (System) Class Loader
Loads classes from the application's classpath (-cp argument or CLASSPATH env variable). This loads your own .class files.
Custom Class Loaders
Developers can create their own class loaders — used in app servers, IDEs, plugin systems, and hot-reload frameworks.

The Sandbox Model

The sandbox model is Java's key concept for untrusted code (originally for applets). Untrusted code runs in a "sandbox" with heavily restricted permissions. The key components enforcing the sandbox are: (1) Class Loader — separates namespaces and prevents untrusted code from replacing trusted classes; (2) Bytecode Verifier — ensures code doesn't violate type safety; (3) Security Manager — intercepts all security-sensitive operations and checks them against the policy. Together, these create an isolated execution environment where foreign code cannot harm the host system.

Applets (the original sandbox use case) are obsolete and removed from modern browsers and JDK. However, the class loader hierarchy and module system (Java 9+) retain sandbox-like namespace isolation.
// Custom Class Loader example
public class MyClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] b = loadClassData(name);   // read bytecode from custom source
        return defineClass(name, b, 0, b.length);  // register with JVM
    }
    private byte[] loadClassData(String name) { /* load from DB, network, etc. */ }
}

2.3 Arrays & Operators

Arrays

Arrays in Java are objects allocated on the heap. They are fixed-size, zero-indexed, and have a length property. Array types: single-dimensional int[], multi-dimensional int[][], and jagged arrays (arrays of arrays with varying sizes).

int[] arr = new int[5];                    // default 0s
int[] arr2 = {10, 20, 30, 40, 50};         // initializer
int[][] matrix = new int[3][4];            // 3 rows, 4 cols
int[][] jagged = new int[3][];
jagged[0] = new int[2];
jagged[1] = new int[5];                   // different sizes
Arrays.sort(arr2);                         // sort array
System.out.println(Arrays.toString(arr2)); // print nicely

Arithmetic Operators

+ (add), - (subtract), * (multiply), / (divide — integer division truncates), % (modulus), ++ (prefix/postfix increment), -- (prefix/postfix decrement). Note: integer division by zero throws ArithmeticException; floating-point division by zero gives Infinity or NaN.

Logical Operators

&& (logical AND — short-circuits), || (logical OR — short-circuits), ! (logical NOT), & (bitwise AND — evaluates both sides), | (bitwise OR), ^ (bitwise XOR), ~ (bitwise complement), << (left shift), >> (arithmetic right shift), >>> (unsigned right shift). Short-circuit evaluation is critical for null-safe checks: obj != null && obj.getValue() > 0.

2.4 Control of Flow

Control Flow Statements

Java control flow statements direct the order of execution. Unlike C, Java's if, while, and for conditions must be boolean (not integer).

// if-else if-else
if (x > 0) { ... } else if (x < 0) { ... } else { ... }

// switch (supports int, char, String, enum since Java 7)
switch (day) {
    case "MON": ...; break;
    default: ...;
}

// for loop
for (int i = 0; i < 10; i++) { ... }

// enhanced for (for-each) — iterates arrays/Iterable
for (String s : list) { ... }

// while and do-while
while (condition) { ... }
do { ... } while (condition);

// Labeled break/continue for nested loops
outer: for (...) {
    inner: for (...) {
        break outer;      // break outer loop
    }
}
2.5 Classes & Instances

Defining and Using Classes

A class is a blueprint. An instance (object) is a concrete realization of that blueprint. Every object has its own copy of instance fields and shares class (static) fields.

public class BankAccount {
    // instance fields
    private String owner;
    private double balance;
    private static int totalAccounts = 0;  // class/static field

    // constructor
    public BankAccount(String owner, double initial) {
        this.owner = owner;
        this.balance = initial;
        totalAccounts++;
    }

    // instance method
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Invalid");
        balance += amount;
    }

    // getter
    public double getBalance() { return balance; }

    // static method
    public static int getTotalAccounts() { return totalAccounts; }
}

2.8 Inheritance, throw/throws, User-defined Exceptions

Inheritance

Inheritance allows a class (subclass) to acquire properties and behaviors of another class (superclass) using extends. Java supports single class inheritance only. The subclass can override, extend, or restrict inherited behavior.

Exception Handling

Java's exception mechanism uses try-catch-finally blocks. The exception hierarchy: Throwable → Error (JVM problems, don't catch) / Exception → RuntimeException (unchecked) / checked exceptions.

throw is used to explicitly throw an exception object. throws is used in the method signature to declare that the method may propagate a checked exception to its caller.

// User-defined exception
class InsufficientFundsException extends Exception {
    private double amount;
    public InsufficientFundsException(double amount) {
        super("Insufficient funds: need " + amount);
        this.amount = amount;
    }
    public double getAmount() { return amount; }
}

class Account {
    // throws declares it to caller; throw actually throws it
    public void withdraw(double amt) throws InsufficientFundsException {
        if (amt > balance) throw new InsufficientFundsException(amt);
        balance -= amt;
    }
}
2.9 StringBuffer, Tokenizer & Applets

StringBuffer

String objects are immutable in Java — every concatenation creates a new object. StringBuffer is a mutable sequence of characters. It is thread-safe (synchronized methods). StringBuilder is the non-synchronized, faster alternative for single-threaded use.

StringBuffer sb = new StringBuffer("Hello");
sb.append(" World");          // "Hello World"
sb.insert(5, ",");             // "Hello, World"
sb.reverse();                  // "dlroW ,olleH"
sb.delete(0, 5);              // " ,olleH" — delete range
String result = sb.toString();

StringTokenizer

java.util.StringTokenizer breaks a string into tokens based on delimiters. It is simpler but less flexible than String.split(). It uses a StreamTokenizer-like approach and is not Unicode-aware for delimiters.

StringTokenizer st = new StringTokenizer("Hello World Java", " ");
while (st.hasMoreTokens()) {
    System.out.println(st.nextToken()); // "Hello", "World", "Java"
}

Unit 03

Threads & AWT

3.1 Creating Threads

Two Ways to Create Threads

A thread is the smallest unit of execution. Java supports multithreading natively. There are two main approaches to create a thread:

// Method 1: Extend Thread class
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread: " + Thread.currentThread().getName());
    }
}
MyThread t = new MyThread();
t.start();       // DO NOT call run() directly — that runs on current thread

// Method 2: Implement Runnable interface (PREFERRED)
Runnable task = () -> System.out.println("Runnable thread");
Thread t2 = new Thread(task, "WorkerThread");
t2.start();

// Java 5+: ExecutorService (preferred for pools)
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> System.out.println("Pool thread"));
pool.shutdown();
Prefer implementing Runnable over extending Thread. It keeps your class free to extend another class, and separates the "what to do" (task) from "how to execute" (thread mechanism).

Thread Priority

Java threads have a priority from 1 (MIN_PRIORITY) to 10 (MAX_PRIORITY), default 5 (NORM_PRIORITY). The thread scheduler uses priority as a hint, but behavior is platform-dependent — never rely on priority alone for correctness. Use t.setPriority(Thread.MAX_PRIORITY).

3.3 Thread Synchronization

Race Conditions & Synchronization

A race condition occurs when multiple threads access shared data concurrently and the result depends on the interleaving of their execution. Java's solution is the monitor (intrinsic lock) — every object has one. The synchronized keyword ensures mutual exclusion.

// Synchronized method — locks on 'this'
public synchronized void increment() {
    count++;
}

// Synchronized block — finer-grained, locks on specified object
public void addItem(Item item) {
    synchronized (listLock) {       // only this critical section is locked
        list.add(item);
    }
    log(item);                      // this runs without lock
}

// Static synchronized — locks on Class object
public static synchronized void staticMethod() { ... }

// java.util.concurrent.locks — more flexible
ReentrantLock lock = new ReentrantLock();
lock.lock();
try { /* critical section */ } finally { lock.unlock(); }  // always unlock
Overriding a synchronized method in a subclass does NOT automatically make the overriding method synchronized. You must explicitly add the synchronized keyword to the overriding method.
3.4 Thread Communication — wait, notify, notifyAll

Inter-Thread Communication

When threads need to coordinate (e.g., producer-consumer), they use wait(), notify(), and notifyAll() defined in Object. These must be called from within a synchronized block/method.

wait()
Releases the monitor lock and suspends the current thread. The thread enters the WAITING state until another thread calls notify() or notifyAll() on the same object. Always call in a loop to handle spurious wakeups.
notify()
Wakes up ONE arbitrarily chosen thread that is waiting on this object's monitor. The awakened thread does not run immediately — it competes for the lock.
notifyAll()
Wakes up ALL threads waiting on this object's monitor. Safer than notify() when multiple threads may be waiting on different conditions.
// Classic Producer-Consumer with wait/notify
class Buffer {
    private int data;
    private boolean hasData = false;

    public synchronized void put(int val) throws InterruptedException {
        while (hasData) wait();    // wait if buffer is full
        data = val; hasData = true;
        notifyAll();               // wake consumer
    }

    public synchronized int get() throws InterruptedException {
        while (!hasData) wait();   // wait if buffer is empty
        hasData = false;
        notifyAll();               // wake producer
        return data;
    }
}
3.5 AWT Components

Abstract Window Toolkit

AWT (java.awt) is Java's original GUI toolkit. It wraps the native OS GUI components. The class hierarchy is: Object → Component → Container → Window → Frame.

Component
The abstract base class for all AWT GUI elements. Provides: size, position, visibility, font, color, event handling. Key subclasses: Button, Label, TextField, TextArea, Checkbox, Choice, List, Canvas, Scrollbar.
Container
A Component that can hold other components. Has a LayoutManager. Subclasses: Panel, Window (→Frame, →Dialog), ScrollPane, Applet.
Frame
A top-level window with a title bar, border, and menu bar. The main window of a standalone Java GUI application.
Panel
A generic container without its own window. Used to group components together with its own layout.
Frame f = new Frame("My App");
f.setSize(400, 300);
Button btn = new Button("Click Me");
TextField tf = new TextField("Enter text", 20);
f.add(btn);
f.add(tf);
f.setVisible(true);
3.6 Layout Managers

Controlling Component Arrangement

Layout Managers implement the LayoutManager interface and automatically arrange components within a container. This is essential for creating GUIs that resize correctly on different screen sizes and OS platforms.

FlowLayout (default for Panel)
Places components in a row, left-to-right. When a row is full, starts a new row. Constructor: new FlowLayout(FlowLayout.CENTER, hgap, vgap)
BorderLayout (default for Frame/Dialog)
5 regions: NORTH, SOUTH, EAST, WEST, CENTER. Each region holds one component. CENTER expands to fill remaining space. f.add(btn, BorderLayout.NORTH)
GridLayout
Divides the container into a grid of equal-sized cells. Each component fills one cell. new GridLayout(rows, cols, hgap, vgap)
CardLayout
Stacks components like a deck of cards. Only one card (component) is visible at a time. Useful for wizard/tabbed-panel style UIs. Methods: show(container, name), next(), previous(), first(), last().
GridBagLayout
The most powerful and complex layout. Uses GridBagConstraints to control: gridx, gridy (cell position), gridwidth, gridheight (cell spanning), weightx, weighty (how extra space is distributed), fill, anchor, insets. Very verbose but allows pixel-precise control.

Insets and Dimensions

Insets(top, left, bottom, right) represents the padding around the edges of a container. Dimension(width, height) encapsulates a component's size. These are used to fine-tune layout spacing.

Adapter Classes

Adapter classes provide empty implementations of listener interfaces with multiple methods (e.g., MouseListener has 5 methods). By extending an Adapter class, you only override the methods you need, reducing boilerplate code. Examples: MouseAdapter, KeyAdapter, WindowAdapter, FocusAdapter.

4.3 Object Serialization

Persisting and Transmitting Objects

Serialization converts an object's state into a byte stream (for saving to disk or sending over a network). Deserialization is the reverse. A class must implement java.io.Serializable (marker interface — no methods). The serialVersionUID is a version identifier for compatibility.

import java.io.*;

class Student implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    int age;
    transient String password;   // NOT serialized
}

// Serialize
try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("student.ser"))) {
    oos.writeObject(new Student("Alice", 20));
}

// Deserialize
try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream("student.ser"))) {
    Student s = (Student) ois.readObject();
    System.out.println(s.name);  // Alice
}
4.4 Sockets & Client-Server Applications

Network Programming with Sockets

Java's java.net package provides TCP (connection-oriented, reliable) and UDP (connectionless, fast) socket support. TCP is used for most client-server applications.

// TCP Server
ServerSocket server = new ServerSocket(8080);
System.out.println("Waiting for clients...");
Socket client = server.accept();      // blocks until client connects
BufferedReader in = new BufferedReader(new InputStreamReader(client.getInputStream()));
PrintWriter out = new PrintWriter(client.getOutputStream(), true);
String msg = in.readLine();
out.println("Echo: " + msg);

// TCP Client
Socket socket = new Socket("localhost", 8080);
PrintWriter out2 = new PrintWriter(socket.getOutputStream(), true);
out2.println("Hello Server!");

Multithreaded Server Design

A single-threaded server can only handle one client at a time. A multithreaded server spawns a new thread (or uses a thread pool) for each client connection, allowing concurrent handling. The main thread just accepts connections in a loop.

ExecutorService pool = Executors.newCachedThreadPool();
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket client = server.accept();
    pool.submit(() -> handleClient(client));  // each client on its own thread
}
4.5 Remote Method Invocation (RMI)

Calling Methods on Remote Objects

RMI allows a Java program to call methods on an object running in another JVM (on a different machine). It provides the illusion of local method calls for remote objects.

Key components: (1) Remote Interface — extends java.rmi.Remote; declares remote methods that throw RemoteException. (2) Remote Object — implements the interface, extends UnicastRemoteObject. (3) RMI Registry — a name service where servers register remote objects and clients look them up. (4) Stub — client-side proxy that marshals arguments and calls the remote object. (5) Skeleton (pre Java 2) — server-side dispatcher (now generated automatically by JVM).

// 1. Define remote interface
public interface Calculator extends Remote {
    int add(int a, int b) throws RemoteException;
}

// 2. Implement remote object
public class CalculatorImpl extends UnicastRemoteObject implements Calculator {
    public CalculatorImpl() throws RemoteException { super(); }
    public int add(int a, int b) { return a + b; }
}

// 3. Register on server
Calculator calc = new CalculatorImpl();
Naming.rebind("rmi://localhost/Calculator", calc);

// 4. Look up and call from client
Calculator c = (Calculator) Naming.lookup("rmi://server/Calculator");
System.out.println(c.add(3, 4));  // 7 — runs on server!

4.7 Collections API

The Java Collections Framework

The Collections Framework provides a unified architecture for representing and manipulating groups of objects. It consists of interfaces, implementations, and algorithms (static methods in Collections and Arrays).

Core Interfaces

Collection<E>
Root interface. Basic operations: add, remove, contains, size, iterator.
List<E>
Ordered (by index), allows duplicates. Key implementations: ArrayList (dynamic array, fast random access), LinkedList (doubly-linked, fast insert/delete).
Set<E>
No duplicates. Implementations: HashSet (unordered, O(1) ops), LinkedHashSet (insertion order), TreeSet (sorted natural order, O(log n)).
Map<K,V>
Key-value pairs. Keys are unique. Implementations: HashMap (unordered, O(1)), LinkedHashMap (insertion/access order), TreeMap (sorted by key), Hashtable (legacy, synchronized).
Queue<E>
FIFO ordering. Implementations: LinkedList, ArrayDeque, PriorityQueue (heap-based, natural order).
Deque<E>
Double-ended queue. Can be used as stack or queue. ArrayDeque preferred over Stack class.

Legacy Classes

Vector
Like ArrayList but synchronized (thread-safe, slower). Legacy class — prefer ArrayList + Collections.synchronizedList() or CopyOnWriteArrayList.
Stack
Extends Vector. Provides push(), pop(), peek(), empty(), search(). Legacy — prefer ArrayDeque.
Hashtable
Like HashMap but synchronized, does not allow null keys/values. Legacy — prefer ConcurrentHashMap.
Enumeration
Legacy iterator-like interface with hasMoreElements() and nextElement(). Replaced by Iterator.
// List — ArrayList
List<String> list = new ArrayList<>();
list.add("Apple"); list.add("Banana"); list.add("Cherry");
list.sort(Comparator.naturalOrder());

// Map — HashMap
Map<String, Integer> freq = new HashMap<>();
freq.put("apple", 3); freq.put("banana", 5);
freq.getOrDefault("mango", 0);              // 0 if missing
freq.forEach((k, v) -> System.out.println(k + ": " + v));

// Set — TreeSet (sorted)
Set<Integer> primes = new TreeSet<>();
primes.addAll(Arrays.asList(5, 3, 7, 2, 11));  // stored sorted: 2,3,5,7,11

// Iterator — universal traversal
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.startsWith("A")) it.remove();   // safe removal during iteration
}

// Utility methods (java.util.Collections)
Collections.sort(list);
Collections.reverse(list);
Collections.shuffle(list);
Collections.unmodifiableList(list);  // read-only wrapper
Java 8+ Stream API (list.stream().filter().map().collect()) provides functional-style operations on collections. Java 9+ factory methods: List.of(1,2,3), Map.of("a",1), Set.of("x","y") create immutable collections.
